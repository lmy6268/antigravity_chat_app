# 개발 블로그 작성 가이드

이 문서는 PR별 개발 블로그 포스트 작성 시 따라야 할 컨벤션을 정의합니다. 우아한형제들 기술 블로그의 형식을 참고하여 작성되었습니다.

## 자동 생성 시점

- **트리거**: Pull Request가 `develop` 브랜치에 머지될 때
- **생성 위치**: GitHub Issues (자동 생성)
- **레이블**: `dev-log`, `auto-generated`, `pr-summary`

## 블로그 글 구조

### 1. 제목

**형식**: `[주제] 핵심 내용: 부제목`

**규칙**:

- 명확하고 구체적으로 작성
- 이모지 사용 지양 (본문에서만 사용)
- 50자 이내 권장
- 독자의 호기심을 자극하는 제목

**예시**:

```
✅ AI와 함께하는 테스트 자동화: 플러그인 개발기
✅ 종단간 암호화 구현기: WebSocket 채팅의 보안 강화
✅ 대시보드 리팩토링: 컴포넌트 분리로 유지보수성 개선
❌ 오늘의 개발 일지
❌ 🚀 코딩했어요
```

### 2. 메타데이터

```markdown
작성일: YYYY. MM. DD.
작성자: [이름]
태그: [주요 기술 스택]
```

### 3. 본문 구조

우아한형제들 기술 블로그의 전형적인 구조를 따릅니다:

#### 프롤로그 (도입부)

- **목적**: 독자의 관심 유도, 배경 설명
- **길이**: 2-3 문단
- **포함 내용**:
  - 문제 상황 또는 배경
  - 왜 이 작업을 하게 되었는지
  - 간단한 결과 미리보기 (선택)

**예시**:

```markdown
## 프롤로그

저희 팀에서 관리하는 레포지토리는 7개입니다. 빠른 비즈니스 과제를 해내가며
필수 테스트코드만 작성하다 보니 커버리지가 부족한 레포지토리가 생겼습니다.
특히 엣지케이스나 예외 상황에 대한 테스트가 부족했고, 이는 운영 안정성을
떨어뜨리는 원인이 되었습니다.
```

#### 주요 섹션 (번호 매기기)

각 섹션은 **숫자로 구분**하고, 논리적 흐름을 따릅니다:

```markdown
## 1. 문제 인식 / 첫 시도

### 1.1 현재 상황 분석

### 1.2 발견한 문제점

## 2. 해결 방법 모색

### 2.1 시도한 방법들

### 2.2 선택한 접근법

## 3. 구현 과정

### 3.1 핵심 구현 내용

### 3.2 기술적 도전과제

## 4. 개선 및 최적화

### 4.1 발견한 문제

### 4.2 개선 방법

## 5. 결과

### 5.1 정량적 성과

### 5.2 정성적 효과

## 6. 맺음말
```

#### 맺음말 (결론)

- **목적**: 핵심 교훈 정리, 향후 계획
- **길이**: 2-4 문단
- **포함 내용**:
  - 핵심 교훈 (불릿 포인트)
  - 최종 성과 요약
  - 향후 계획 또는 개선 방향

**예시**:

```markdown
## 맺음말

"AI가 테스트 코드를 자동으로 작성해준다면?"

이 질문에 대한 우리의 답은 "완벽한 자동화는 불가능하지만,
효율적인 보조는 가능하다"입니다.

**핵심 교훈**

- **완벽한 자동화는 환상**: AI는 "보조 도구"이지 "대체 도구"가 아닙니다.
- **도구 간 역할 분담**: 각 도구의 강점을 활용한 협업이 중요합니다.
- **팀과 함께 개선**: 혼자 만들지 말고 팀과 함께 사용하며 개선합니다.
```

## 작성 스타일 가이드

### 문체

- **친근하고 읽기 쉬운 문체** 사용
- **1인칭 복수형** 사용 ("저희", "우리")
- **구어체와 문어체의 적절한 조화**
- 독자에게 말하듯이 작성

### 기술 용어

- 처음 등장 시 **간단한 설명 추가**
- 약어는 **풀어서 설명** (예: E2EE → 종단간 암호화)
- 복잡한 개념은 **비유나 예시** 활용

### 코드 블록

```markdown
// 좋은 예: 설명 + 코드 + 결과
**문제 상황**

\`\`\`kotlin
// 기존 코드 - 타입 불일치 오류
every { location.latitude } returns "37.5" // Double인데 String
\`\`\`

**해결 방법**

\`\`\`kotlin
// 수정된 코드
every { location.latitude } returns 37.5 // 올바른 타입
\`\`\`
```

### 시각적 요소

#### 표 활용

비교, 통계, 결과를 표로 정리:

```markdown
| 항목      | Before | After | 개선율 |
| --------- | ------ | ----- | ------ |
| 작업 시간 | 10분   | 3분   | 70%    |
| 커버리지  | 67%    | 95%   | +28%p  |
```

#### 강조 표시

```markdown
**중요한 내용**은 볼드 처리
`코드나 파일명`은 백틱 사용

> 인용구는 blockquote 사용
```

#### 이모지 활용

**적절한 위치에만 사용**:

```markdown
✅ 성공 케이스
❌ 실패 케이스
📊 통계/데이터
🔍 분석/발견
💡 아이디어/팁
⚠️ 주의사항
```

### 수치 데이터

**구체적인 수치**로 신뢰도 향상:

```markdown
❌ "많은 시간이 절약되었습니다"
✅ "클래스당 작업 시간 70% 단축 (10분 → 3분)"

❌ "커버리지가 증가했습니다"
✅ "커버리지 67% → 95% (+28%p)"
```

### 문제-해결 구조

각 섹션에서 **문제 → 시도 → 해결** 흐름 유지:

```markdown
### 3.1 발견한 문제

**문제 1: 기존 테스트가 사라진다**

가장 심각한 문제였습니다. 테스트를 추가하려고 했더니
기존 테스트가 완전히 대체되어 버렸습니다.

\`\`\`kotlin
// 기존 테스트 (163줄)
class ProductServiceTest : FunSpec({ ... })

// AI 생성 결과 - 기존 테스트 완전 교체
class ProductServiceTest : FunSpec({
// 기존 테스트는 사라짐...
})
\`\`\`

**해결 방법**

템플릿 방식으로 전환하여 기존 테스트를 보존하면서
새로운 테스트를 추가하는 방식으로 개선했습니다.
```

## PR별 개발 블로그 특화 가이드

### 제목 형식

```
[날짜] PR #[번호]: [핵심 작업 요약]
```

**예시**:

```
2025.12.24 PR #42: E2EE 방 초대 기능 구현
2025.12.23 PR #38: 대시보드 컴포넌트 리팩토링
```

### 구조 (PR 요약 버전)

PR 블로그는 해당 PR의 변경사항을 중심으로 작성:

```markdown
## 프롤로그

이번 PR의 배경과 목적 설명

## 1. 문제 인식 / 배경

### 1.1 현재 상황

### 1.2 해결하고자 한 문제

## 2. 해결 방법

### 2.1 설계 및 접근법

### 2.2 주요 구현 내용

## 3. 구현 상세

### 3.1 핵심 변경사항

### 3.2 기술적 도전과제

## 4. 결과

### 4.1 정량적 성과

### 4.2 개선 효과

## 맺음말

핵심 교훈과 향후 계획
```

### PR 통계 정보 포함

```markdown
## 📊 PR 통계

| 항목      | 수치      |
| --------- | --------- |
| PR 번호   | #42       |
| 작성자    | @username |
| 커밋 수   | 8개       |
| 변경 파일 | 15개      |
| 추가된 줄 | +847      |
| 삭제된 줄 | -234      |

**🔗 링크**

- [PR 보기](https://github.com/repo/pull/42)
```

## 체크리스트

글 작성 후 다음 항목을 확인하세요:

### 구조

- [ ] 제목이 명확하고 구체적인가?
- [ ] 프롤로그에서 배경을 잘 설명했는가?
- [ ] 섹션이 논리적 순서로 구성되었는가?
- [ ] 맺음말에서 핵심을 잘 정리했는가?

### 내용

- [ ] 기술 용어에 설명을 추가했는가?
- [ ] 코드 블록에 주석이나 설명이 있는가?
- [ ] 구체적인 수치 데이터를 포함했는가?
- [ ] 문제-해결 구조가 명확한가?

### 가독성

- [ ] 문단이 너무 길지 않은가? (3-5줄 권장)
- [ ] 표나 리스트를 적절히 활용했는가?
- [ ] 이모지를 과도하게 사용하지 않았는가?
- [ ] 오타나 문법 오류가 없는가?

### 기술적 정확성

- [ ] 코드가 실제로 동작하는가?
- [ ] 기술 스택 정보가 정확한가?
- [ ] 성과 수치가 검증 가능한가?
